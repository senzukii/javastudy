Patterns de Structure Centrés sur la construction de structures à base d'objets
Composite
"Le Composite compose des objets en structures arborescentes pour représenter des hiérarchies composant/composé. Il permet aux clients de traiter de la même façon les objets individuels et les combinaisons."

🔴 Problème résolu

Quand on a une structure arborescente de profondeur inconnue (boîtes dans des boîtes, dossiers dans des dossiers), on est obligé d'écrire deux traitements différents : un pour les feuilles, un pour les composés. Le code client devient complexe.

✅ Solution

On définit une interface commune à la feuille et au composé. Le client appelle toujours la même méthode, et la récursivité fait le reste — en descendant jusqu'aux feuilles puis en remontant.
ParticipantRôleExemple Amazon
ComposantInterface commune — définit l'opérationItem (avec getPrice())
FeuilleObjet concret sans enfants — implémente directementProduit
ComposéContient une liste de Composants — délègue récursivementBox
ClientManipule tout via l'interface ComposantSystème de calcul du prix
Diagramme général
Exemple concret — Amazon
Séquence d'exécution — Box_A( Produit_1(10€) + Produit_2(20€) + Box_B( Produit_3(15€) ) )
Le Client appelle une seule méthode sur la racine — la récursivité descend jusqu'aux feuilles et remonte en accumulant le résultat.
Décorateur
"Le Décorateur attache dynamiquement des responsabilités supplémentaires à un objet. Il fournit une alternative souple à la création de sous-classes pour étendre les fonctionnalités."

🔴 Problème résolu

Si on veut enrichir un objet avec des options, l'héritage mène à une explosion combinatoire de sous-classes. Avec N décorateurs possibles, on aurait 2^N sous-classes. Ex : CaféLait, CaféChantilly, CaféLaitChantilly...

✅ Solution

On "enveloppe" un objet dans un Décorateur qui ajoute son comportement avant/après avoir délégué à l'objet qu'il contient. Comme des poupées russes — chaque enveloppe ajoute quelque chose.
ParticipantRôleExemple Café
ComposantAbstraitInterface commune — définit l'opérationBoisson (getCost())
ComposantConcretL'objet de base à décorerCafé (1.00€)
DécorateurAbstrait — hérite ET contient un ComposantAbstraitCondimentDecorateur
DécorateurConcretAjoute son comportement + délègue via superLait (+0.20€), Chantilly (+0.50€)
Diagramme général
Exemple concret — Café
Séquence d'exécution — Café + Lait + Chantilly
Le Décorateur hérite ET contient le ComposantAbstrait — héritage pour être interchangeable, référence pour envelopper et déléguer.
Patterns de Comportement Centrés sur les interactions entre objets
Strategy
"Le Strategy définit une famille d'algorithmes, encapsule chacun d'eux et les rend interchangeables. Il permet de faire varier l'algorithme indépendamment des clients qui l'utilisent."

🔴 Problème résolu

Sans Strategy, les différents algorithmes sont dans un if/else dans la classe qui les utilise. Ajouter un algorithme oblige à modifier cette classe, violant Open/Closed.

✅ Solution

Chaque algorithme est encapsulé dans sa propre classe (ConcreteStrategy). Le Context les utilise via une interface commune — il ne sait jamais quel algorithme s'exécute concrètement.
ParticipantRôleExemple Paiement
StrategyInterface commune — définit le contrat de l'algorithmePaiementStrategy
ConcreteStrategyImplémentation concrète d'un algorithmeBancontact, Visa, Mastercard
ContextInterface entre le client et l'algorithme — délègueSystemePaiement
ClientChoisit la stratégie et l'injecte dans le ContextL'application
Diagramme général
Exemple concret — Paiement
Séquence d'exécution
Le Context est l'interface entre le client et l'algorithme — il reçoit la demande et délègue à la Strategy sans savoir laquelle s'exécute.
Observer — Mode Push
"Le pattern Observer définit une relation un-à-plusieurs : quand un objet change d'état, tous ceux qui en dépendent sont notifiés automatiquement. En mode Push, le Sujet envoie les données directement aux observateurs."

🔴 Problème résolu

Sans Observer, les observateurs devraient vérifier manuellement et périodiquement si le sujet a changé — coûteux et inefficace. Le Sujet devrait connaître concrètement chaque observateur pour les notifier.
ParticipantRôleExemple Météo
SujetClasse abstraite — gère la liste, notifieMétéo
SujetConcretPossède l'état surveilléOrageux
ObservateurInterface — définit actualise()Citoyen
ObservateurConcretRéagit à la notification avec les données reçuesJeanMichel
Diagramme général
Exemple — Météo Push
Séquence d'exécution — Mode Push
En mode Push, le Sujet envoie tout — l'Observateur reçoit les données directement dans l'appel à actualise(), sans avoir à les redemander.
Observer — Mode Pull
"En mode Pull, le Sujet notifie simplement que son état a changé, et chaque Observateur va chercher lui-même les données dont il a besoin via une référence vers le Sujet."
Push vs PullPushPull
Appel actualise()actualise(données)actualise() — vide
DonnéesEnvoyées par le SujetL'Observateur va les chercher
Référence SujetNon nécessaireObservateurConcret connaît SujetConcret
AvantageSimple, directL'observateur prend uniquement ce dont il a besoin
InconvénientEnvoie des données inutilesCouplage plus fort (connaît le SujetConcret)
Diagramme Pull — ajout de la référence
Exemple — Météo Pull
Séquence d'exécution — Mode Pull
En mode Pull, le Sujet dit juste "quelque chose a changé" — c'est l'Observateur qui décide quelles données il veut récupérer via getEtat().
Patterns de Création Centrés sur la création et la configuration d'objets
Simple Factory
"Le Simple Factory centralise la création d'objets dans une seule classe, évitant au client de savoir quelle classe concrète instancier."

🔴 Problème résolu

Sans Factory, le code client contient des if/else éparpillés partout pour instancier des objets. Ajouter un nouveau type d'objet oblige à modifier plusieurs endroits du code. 
if (type == "margherita") pizza = new Margherita();
else if (type == "regina") pizza = new Regina();
// → modification à faire partout !

✅ Solution

On centralise toute la logique de création dans une unique classe Fabrique. Le client demande un objet sans savoir quelle classe concrète est instanciée.

Participants

ParticipantRôleExemple Pizza
ProduitInterface commune à tous les produitsPizza
ProduitConcretImplémentation concrèteMargherita, Regina, Calzone
FabriqueCentralise la création — méthode getInstance()PizzeriaFactory
ClientDemande un objet sans savoir lequelLe client qui commande
Diagramme général
Exemple concret — Pizzeria
Séquence d'exécution — commande d'une pizza

⚠️ Limite — violation Open/Closed

Ajouter un nouveau type de pizza oblige à modifier la classe PizzeriaFactory. C'est pourquoi on a inventé le Factory Method.
La Fabrique connaît tous les produits, le Client ne connaît que l'interface — il ne sait jamais quelle classe concrète est instanciée.
Factory Method
"Le Factory Method définit une interface pour créer un objet, mais laisse les sous-classes décider quelle classe instancier."

🔴 Problème résolu

Le Simple Factory viole le principe Open/Closed — chaque nouveau produit implique de modifier la Fabrique. De plus, si on a plusieurs contextes (plusieurs pizzerias), on a un if/else monstrueux avec ville + type.

✅ Solution

On déclare la méthode de création dans une classe abstraite (le Créateur), et chaque sous-classe (CréateurConcret) l'implémente à sa façon. Ajouter une nouvelle pizzeria = créer une nouvelle sous-classe, sans toucher au code existant.
ParticipantRôleExemple Pizza
CréateurClasse abstraite — déclare creerProduit()Pizzeria (abstraite)
CréateurConcretImplémente creerProduit() pour son contextePizzeriaCharleroi, PizzeriaBruxelles
ProduitInterface communePizza
ProduitConcretPizza spécifique à un contexteMargheritaCharleroi, MargheritaBruxelles
Diagramme général
Exemple concret — Pizzerias
Séquence d'exécution
Chaque CréateurConcret est responsable de ses propres produits — ajouter une nouvelle pizzeria ne touche à rien d'existant.
Abstract Factory
"L'Abstract Factory fournit une interface pour créer des familles d'objets liés ou dépendants sans préciser leur classe concrète."

🔴 Problème résolu

Le Factory Method crée un seul type de produit. Mais que faire quand on a besoin de créer une famille de produits cohérents entre eux ? Une pizzeria italienne ne doit pas mélanger des ingrédients italiens et belges !

✅ Solution

L'Abstract Factory déclare une méthode de création par type de produit dans la famille. Chaque FabriqueConcrete implémente toutes ces méthodes pour garantir la cohérence de la famille.
ParticipantRôleExemple Pizza
FabriqueAbstraiteInterface avec une méthode par produit de la familleIngredientsFactory
FabriqueConcreteCrée une famille cohérente de produitsIngredientsItaliens, IngredientsBelges
ProduitAbstraitInterface pour chaque type de produitPate, Sauce, Fromage
ProduitConcretImplémentation spécifiquePateFine, SauceTomate, Mozzarella
ClientUtilise la fabrique sans connaître les classes concrètesLa Pizzeria
Diagramme général
Exemple concret — Ingrédients pizza
Séquence d'exécution
L'Abstract Factory garantit la cohérence d'une famille — une fabrique italienne ne produit que des ingrédients italiens, jamais un mélange.

↳ Comparaison des 3 patterns Factory

Simple Factory

Une fabrique, tous les produits

Centralise la création mais viole Open/Closed — ajouter un produit = modifier la fabrique.


creerProduit(String type)
Factory Method

Une sous-classe, un produit

Chaque sous-classe crée son propre produit. Respecte Open/Closed — ajouter un créateur sans toucher au code.


abstract creerProduit()
Abstract Factory

Une sous-classe, une famille

Chaque sous-classe crée une famille complète et cohérente de produits liés.


creerA() creerB() creerC()
Builder
"Le Builder permet de construire des objets complexes étape par étape, en séparant la construction de l'objet de sa représentation finale."

🔴 Problème résolu

Certains objets sont tellement complexes qu'un constructeur devient illisible avec de nombreux paramètres. De plus, l'ordre des étapes de construction peut être important. 
new Burger("complet","poulet","salade","tomate","mayo","ketchup");

✅ Solution

On sépare la construction en étapes. Le Directeur orchestre l'ordre des étapes, le BuilderConcret sait comment réaliser chaque étape.
ParticipantRôleExemple Burger
BuilderInterface — définit les étapes de constructionBurgerBuilder
BuilderConcretImplémente chaque étape concrètementBurgerPoulet, BurgerBoeuf
DirecteurOrchestre les étapes dans le bon ordreCuisinier
ProduitL'objet complexe résultantBurger
Diagramme général
Exemple concret — Burger
Séquence d'exécution
Le Directeur connaît l'ordre des étapes, le BuilderConcret connaît comment réaliser chaque étape.
