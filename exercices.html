Création Creational Patterns
Simple Factory Scénario : Garage de véhicules Création
Implémentation Java
interface Vehicule {
    void demarrer();
}

class Voiture  Vehicule {
    public void demarrer() { System.out.println("Vroooom"); }
}
class Moto implements Vehicule {
    public void demarrer() { System.out.println("Pétarade"); }
}

class FabriqueVehicule {
    // Méthode statique — pas d'instance de FabriqueVehicule nécessaire
    public  Vehicule creer(String type) {
        switch (type) {
            case "voiture": return new ();
            case "moto"   : return new Moto();
            default: throw new IllegalArgumentException("Type inconnu: " + type);
        }
    }
}

// Utilisation — appel statique, sans new FabriqueVehicule()
Vehicule v = FabriqueVehicule.("voiture");
v.demarrer(); // Vroooom
Diagramme UML
┌────────────────────────────────┐ │ FabriqueVehicule │ │ + ① creer(type): Vehicule │ └────────────────┬───────────────┘ │ «crée» ▼ ┌───────────────────┐ │ <<interface>> │ │ ② Vehicule │ │ + demarrer() │ └─────────┬─────────┘ │ ③ ┌────────┼────────┐ ▼ ▼ ▼ Voiture Moto Camion
① Modificateur de la méthode creer()
② Nom de l'interface produit
③ Type de relation (Voiture/Moto/Camion → Vehicule)
Factory Method Scénario : Boulangeries régionales Création
Implémentation Java
interface Pain {
    void preparer();
    void emballer();
}
class Baguette implements Pain {
    public void preparer() { System.out.println("Façonnage baguette"); }
    public void emballer() { System.out.println("Papier baguette"); }
}
class Brioche implements Pain {
    public void preparer() { System.out.println("Pétrissage brioche"); }
    public void emballer() { System.out.println("Boîte brioche"); }
}

 class Boulangerie {
    // Factory Method — chaque sous-classe décide quel Pain créer
    protected  Pain creerPain();

    // Méthode template — utilise le factory method
    public void vendre() {
        Pain pain = ();
        pain.preparer();
        pain.emballer();
    }
}

class BoulangerieParisienne  Boulangerie {
    @Override
    protected Pain creerPain() { return new Baguette(); }
}

class BoulangerieLyonnaise extends Boulangerie {
    
    protected Pain creerPain() { return new Brioche(); }
}

Abstract Factory Scénario : Thèmes UI (Windows / Mac) Création
Implémentation Java
interface Bouton   { void afficher(); }
interface Checkbox { void cocher(); }

// Famille Windows
class BoutonWindows   implements Bouton   { public void afficher() { System.out.println("[OK]"); } }
class CheckboxWindows implements Checkbox { public void cocher()   { System.out.println("[x]"); } }

// Famille Mac
class BoutonMac   implements Bouton   { public void afficher() { System.out.println("(OK)"); } }
class CheckboxMac implements Checkbox { public void cocher()   { System.out.println("(✓)"); } }

// Fabrique abstraite — déclare la famille de produits
interface  {
       creerBouton();
    Checkbox creerCheckbox();
}

// Fabrique concrète Windows
class FabriqueWindows  FabriqueUI {
    public Bouton   creerBouton()   { return new BoutonWindows(); }
    public Checkbox creerCheckbox() { return new CheckboxWindows(); }
}

// Fabrique concrète Mac
class FabriqueMac implements FabriqueUI {
    public Bouton   creerBouton()   { return new BoutonMac(); }
    public Checkbox creerCheckbox() { return new CheckboxMac(); }
}

// Client — dépend uniquement de l'interface
class Application {
    private  fabrique;

    public Application(FabriqueUI fabrique) {
        this.fabrique = fabrique;
    }

    public void construire() {
        Bouton   btn = fabrique.creerBouton();
        Checkbox cb  = fabrique.creerCheckbox();
        btn.afficher();
        cb.cocher();
    }
}
Diagramme UML
┌──────────────────────────────┐ │ <<interface>> ① │ │ + creerBouton(): Bouton │ │ + creerCheckbox(): Checkbox │ └──────────────────────────────┘ ▲ ───────┴──────── │ │ │ ② │ ② ┌───────────┴───┐ ┌─────┴───────────┐ │FabriqueWindows│ │ FabriqueMac │ └───────────────┘ └─────────────────┘ Application ──③──→ FabriqueUI (dépendance)
① Nom de l'interface fabrique abstraite
② Relation des fabriques concrètes → FabriqueUI
③ Nature du lien Application → FabriqueUI
Builder Scénario : Constructeur de requêtes SQL Création
Implémentation Java
class RequeteSQL {
    private final String table;
    private final String condition;
    private final int limite;

    private RequeteSQL(Builder builder) {
        this.table     = builder.table;
        this.condition = builder.condition;
        this.limite    = builder.limite;
    }

    public static class  {
        private String table;
        private String condition = "";
        private int    limite    = 100;

        public Builder table(String t) {
            this.table = t;
            return ; // méthode fluente
        }

        public Builder where(String c) {
            this.condition = c;
            return this;
        }

        public Builder limite(int l) {
            this.limite = l;
            return this;
        }

        public RequeteSQL () {
            return new RequeteSQL(this);
        }
    }

    @Override
    public String toString() {
        return "SELECT * FROM " + table
            + (condition.isEmpty() ? "" : " WHERE " + condition)
            + " LIMIT " + limite;
    }
}

// Utilisation — chaînage fluent
RequeteSQL req = new RequeteSQL.()
    .table("clients")
    .where("age > 18")
    .limite(50)
    .();
System.out.println(req); // SELECT * FROM clients WHERE age > 18 LIMIT 50
Diagramme UML
┌──────────────────────────────────┐ │ RequeteSQL │ │ - table, condition, limite │ │ - RequeteSQL(Builder) │ │ │ │ ┌──────────────────────────┐ │ │ │ «inner class» ① │ │ │ │ + table(t): Builder │ │ │ │ + where(c): Builder │ │ │ │ + limite(l): Builder │ │ │ │ + ②(): RequeteSQL │ │ │ └──────────────────────────┘ │ └──────────────────────────────────┘ Chaque méthode retourne ③ pour le chaînage.
① Nom de la classe interne
② Méthode terminale qui crée le produit final
③ Valeur retournée par chaque setter (fluent)

Composite Scénario : Système de fichiers Structure
Implémentation Java
// Interface commune — Fichier et Dossier traités de façon identique
interface  {
    int getTaille();
}

// Feuille — pas d'enfants
class Fichier implements Composant {
    private String nom;
    private int taille;

    public Fichier(String nom, int taille) {
        this.nom    = nom;
        this.taille = taille;
    }

    @Override
    public int getTaille() { return taille; }
}

// Composite — contient d'autres Composant
class Dossier  Composant {
    private String nom;
    private List<> enfants = new ArrayList<>();

    public void ajouter(Composant c) {
        .add(c);
    }

    @Override
    public int getTaille() {
        return enfants.stream()
            .(Composant::getTaille)
            .sum();
    }
}

// Utilisation
Dossier racine = new Dossier("racine");
racine.ajouter(new Fichier("readme.txt", 2));
racine.ajouter(new Fichier("app.jar",  150));
System.out.println(racine.getTaille()); // 152
Diagramme UML
┌────────────────────┐ │ <<interface>> │ │ ① Composant │ │ + getTaille(): int │ └────────────────────┘ ▲ ────────┴──────── │ ② │ ② ┌───────┴──────┐ ┌─────┴──────────────────┐ │ Fichier │ │ Dossier │ │ - taille:int │ │ - enfants: List<③> │ └──────────────┘ │ + ajouter(Composant) │ └──────────────┬──────────┘ │ ④ 0..* ◆ (contient)
① Nom de l'interface commune
② Relation Fichier/Dossier → interface
③ Type générique de la liste d'enfants
④ Multiplicité : un Dossier contient combien de Composant ?
Décorateur Scénario : Café avec extras Structure
Implémentation Java
interface Boisson {
    String getDescription();
    double getPrix();
}

// Composant concret
class Cafe  Boisson {
    public String getDescription() { return "Café"; }
    public double getPrix()        { return 1.50; }
}

// Décorateur abstrait — implémente ET contient Boisson
abstract class DecorateurBoisson  Boisson {
    protected Boisson ; // délégué

    public DecorateurBoisson(Boisson b) { this.boisson = b; }

    public String getDescription() { return boisson.getDescription(); }
    public double getPrix()        { return boisson.getPrix(); }
}

// Décorateurs concrets
class Lait extends DecorateurBoisson {
    public Lait(Boisson b) { super(b); }

    @Override
    public String getDescription() {
        return .getDescription() + " + Lait";
    }

    @Override
    public double getPrix() {
        return .getPrix() + 0.30;
    }
}

class Sucre extends DecorateurBoisson {
    public Sucre(Boisson b) { super(b); }
    public String getDescription() { return boisson.getDescription() + " + Sucre"; }
    public double getPrix()        { return boisson.getPrix() + 0.10; }
}

// Utilisation — empilement de décorateurs
Boisson commande = new Sucre(new Lait(new Cafe()));
System.out.println(commande.getDescription()); // Café + Lait + Sucre
System.out.println(commande.getPrix());        // 1.90
Diagramme UML
┌──────────────────────────┐ │ <<interface>> │ │ ① │ │ + getDescription() │ │ + getPrix(): double │ └──────────┬───────────────┘ ▲ │ ─────┤ │ «②» (has-a) │ │ ▼ │ ┌─┴─────────────────────┐ │ │ abstract │ │ │ DecorateurBoisson │ │ │ # boisson: ③ │ │ └──────────┬────────────┘ │ │ ④ │ ───────┴────── │ │ │ │ ┌───┴──┐ ┌───┴───┐ │ │ Lait │ │ Sucre │ │ └──────┘ └───────┘ │ └──── Cafe
① Interface Component (contrat commun)
② Nature du lien DecorateurBoisson → interface (composition)
③ Type du champ délégué dans DecorateurBoisson
④ Relation Lait/Sucre → DecorateurBoisson
Comportement Behavioral Patterns
Strategy Scénario : Modes de paiement Comportement
Implémentation Java
// Strategy — contrat commun pour tous les algorithmes
interface  {
    void payer(double montant);
}

// Stratégies concrètes
class PaiementCarte  StrategiePaiement {
    public void payer(double montant) {
        System.out.println("Paiement carte: " + montant + "€");
    }
}

class PaiementPayPal implements StrategiePaiement {
    public void payer(double montant) {
        System.out.println("PayPal: " + montant + "€");
    }
}

class PaiementCrypto implements StrategiePaiement {
    public void payer(double montant) {
        System.out.println("Crypto ≈ " + (montant / 45000) + " BTC");
    }
}

// Contexte — délègue à la stratégie active
class Caisse {
    private StrategiePaiement ;

    public void setStrategie(StrategiePaiement s) {
        this. = s;
    }

    public void effectuerPaiement(double montant) {
        .payer(montant);
    }
}

// Utilisation — on peut changer de stratégie à chaud
Caisse caisse = new Caisse();
caisse.setStrategie(new PaiementCarte());
caisse.effectuerPaiement(49.99); // Paiement carte: 49.99€

caisse.setStrategie(new PaiementPayPal());
caisse.effectuerPaiement(49.99); // PayPal: 49.99€

┌─────────────────────┐ ┌──────────────────────┐ │ «Context» │ │ <<interface>> │ │ Caisse │──①──→ │ ② │ │ - strategie │ │ + payer(montant) │ │ + setStrategie() │ └──────────────┬────────┘ │ + effectuerPaiement│ ▲ └─────────────────────┘ ──────────┴────────── │ │ │ ┌───────┴──┐ ┌───┴────┐ ┌──┴──────┐ │ Paiement │ │Paiement│ │Paiement │ │ Carte │ │PayPal │ │ Crypto │ └──────────┘ └────────┘ └─────────┘ ③ (relation commune)

② Nom de l'interface Strategy
③ Relation des stratégies concrètes → interface
Observer — Push Scénario : Station météo Comportement
Implémentation Java
// PUSH : le Subject envoie les données dans l'appel update()
interface Observateur {
    // La donnée est POUSSÉE en paramètre
    void mettreAJour( temperature);
}

class StationMeteo {
    private List<Observateur> observateurs = new ArrayList<>();
    private double temperature;

    public void abonner( o) {
        observateurs.add(o);
    }

    public void setTemperature(double t) {
        this.temperature = t;
        (); // déclenche la notification
    }

    private void notifier() {
        for (Observateur o : observateurs) {
            o.mettreAJour(); // donnée poussée
        }
    }
}

// Observer concret — reçoit la donnée sans interroger le Subject
class AfficheurMeteo implements Observateur {
    @Override
    public void mettreAJour(double temperature) {
        System.out.println("Météo : " + temperature + "°C");
    }
}

// Utilisation
StationMeteo station = new StationMeteo();
station.abonner(new AfficheurMeteo());
station.setTemperature(22.5); // → "Météo : 22.5°C"
Diagramme UML
┌──────────────────────────────┐ ┌─────────────────────────────┐ │ «Subject» ① │ │ <<interface>> │ │ StationMeteo │──②──→│ Observateur │ │ - observateurs: List │ │ + mettreAJour(③ t) │ │ + abonner(Observateur) │ └─────────────────────────────┘ │ + setTemperature(double) │ ▲ └──────────────────────────────┘ │ implements AfficheurMeteo (reçoit la donnée)
① Rôle UML de StationMeteo dans le pattern
② Nature de la relation Subject → Observateur
③ Type du paramètre dans mettreAJour() — ce qui est POUSSÉ
Observer — Pull Scénario : Cours de bourse Comportement
Implémentation Java
// PULL : le Subject ne pousse rien — l'Observer interroge lui-même
interface Observateur {
    // Pas de paramètre — l'observer ira chercher les données
    void mettreAJour(); // laisser vide — pas de paramètre (différence clé avec Push)
}

class Bourse {
    private List<Observateur> observateurs = new ArrayList<>();
    private double cours;

    // Getter exposé pour que les observers puissent TIRER la donnée
    public double getCours() { return cours; }

    public void setCours(double c) {
        this.cours = c;
        notifier();
    }

    private void notifier() {
        for (Observateur o : observateurs) {
            o.mettreAJour(); // aucune donnée transmise
        }
    }
}

// Observer concret — stocke une référence au Subject pour l'interroger
class Investisseur implements Observateur {
    private  bourse; // référence au Subject

    public Investisseur(Bourse b) { this.bourse = b; }

    @Override
    public void mettreAJour() {
        // PULL : c'est l'observer qui va chercher la donnée
        double cours = .();
        System.out.println("Cours actuel : " + cours + "€");
    }
}

// Utilisation
Bourse cac40 = new Bourse();
Investisseur inv = new Investisseur(cac40);
cac40.abonner(inv);
cac40.setCours(7240.50); // → "Cours actuel : 7240.5€"
Push vs Pull : Push → mettreAJour(double data) — la donnée arrive dans l'appel.
Pull → mettreAJour() sans paramètre — l'observer stocke une référence au Subject et appelle son getter.
Diagramme UML
┌────────────────────────────┐ ┌──────────────────────────┐ │ «Subject» Bourse │ │ <<interface>> │ │ + ①(): double │ │ Observateur │ │ + setCours(double) │──②──→│ + mettreAJour③ │ └────────────────────────────┘ └──────────────────────────┘ ▲ ▲ │ stocke référence │ implements │ Investisseur Investisseur ──────────────────────────────────┘ (appelle getCours() pour récupérer la donnée)
① Getter exposé par le Subject pour le PULL
② Relation Subject → Observateur
③ Signature de mettreAJour — différence clé avec le Push
