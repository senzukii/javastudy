Principes SOLID

Les 5 principes fondamentaux de la conception orientée objet — pour un code maintenable, extensible et robuste.

S
Single Responsibility Principle
Principe de responsabilité unique
"Une classe ne doit avoir qu'une seule raison de changer."

Chaque classe doit avoir une seule responsabilité. Si une classe fait trop de choses, elle devient fragile — modifier une fonctionnalité risque d'en casser une autre.

❌ Violation SRP
public class Employe {
  // Responsabilité 1 : données
  private String nom;
  private double salaire;

  // Responsabilité 2 : calcul
  public double calculerSalaire() { ... }

  // Responsabilité 3 : persistance
  public void sauvegarderEnBDD() { ... }

  // Responsabilité 4 : rapport
  public String genererRapport() { ... }
  // → 4 raisons de changer !
✅ Respect SRP
public class Employe {
  private String nom;
  private double salaire;
}

public class CalculateurSalaire {
  public double calculer(Employe e) { ... }
}

public class EmployeRepository {
  public void sauvegarder(Employe e) { ... }
}

public class RapportEmploye {
  public String generer(Employe e) { ... }
}
// → chaque classe = 1 raison de changer

✅ Pourquoi c'est important

Une classe avec une seule responsabilité est plus facile à tester, à modifier et à comprendre. Si le mode de calcul du salaire change, seul CalculateurSalaire est modifié — pas le reste.
Une classe = une responsabilité = une seule raison de changer.
O
Open/Closed Principle
Principe ouvert/fermé
"Une classe doit être ouverte à l'extension mais fermée à la modification."

On doit pouvoir ajouter de nouvelles fonctionnalités sans modifier le code existant. On étend via l'héritage ou la composition — pas en changeant ce qui marche déjà.

❌ Violation OCP
public class Calculateur {
  public double calculerPrime(Employe e) {
    if (e.getType() == "dev") {
      return e.getSalaire() * 0.1;
    } else if (e.getType() == "manager") {
      return e.getSalaire() * 0.2;
    }
    // Ajouter un type = modifier ici !
  }
}
✅ Respect OCP
public interface CalculateurPrime {
  double calculer(Employe e);
}

public class PrimeDev implements CalculateurPrime {
  public double calculer(Employe e) {
    return e.getSalaire() * 0.1;
  }
}

public class PrimeManager implements CalculateurPrime {
  public double calculer(Employe e) {
    return e.getSalaire() * 0.2;
  }
}
// Ajouter un type = nouvelle classe !

✅ Lien avec les Design Patterns

C'est exactement ce que respectent Factory Method (ajouter un créateur sans modifier la hiérarchie), Strategy (ajouter un algorithme sans modifier le Context) et Observer (ajouter un observateur sans modifier le Sujet).
Ouverte à l'extension (nouvelles classes), fermée à la modification (code existant intouché).
L
Liskov Substitution Principle
Principe de substitution de Liskov
"Les objets d'une classe dérivée doivent pouvoir remplacer les objets de la classe de base sans altérer le bon fonctionnement du programme."

Si B hérite de A, alors partout où on utilise A, on doit pouvoir utiliser B sans que ça plante. La sous-classe ne doit pas affaiblir les garanties de la classe parente.

❌ Violation LSP
public class Rectangle {
  protected int largeur, hauteur;
  public void setLargeur(int l) { largeur = l; }
  public void setHauteur(int h) { hauteur = h; }
  public int getSurface() { return largeur * hauteur; }
}

public class Carre extends Rectangle {
  // Problème : un carré doit avoir largeur = hauteur
  public void setLargeur(int l) {
    largeur = hauteur = l; // casse le comportement !
  }
}
// rectangle.setLargeur(5); rectangle.setHauteur(4);
// Si c'est un Carré → surface = 16, pas 20 !
✅ Respect LSP
public interface Forme {
  int getSurface();
}

public class Rectangle implements Forme {
  private int largeur, hauteur;
  public int getSurface() {
    return largeur * hauteur;
  }
}

public class Carre implements Forme {
  private int cote;
  public int getSurface() {
    return cote * cote;
  }
}
// Pas d'héritage problématique !

⚠️ Comment détecter une violation LSP

Si tu dois écrire des instanceof ou des if pour vérifier le type réel d'un objet avant de l'utiliser, tu violes probablement LSP. La sous-classe ne peut pas être substituée tranquillement.
Une sous-classe peut toujours remplacer sa classe parente sans surprises — le comportement attendu est préservé.
I
Interface Segregation Principle
Principe de ségrégation des interfaces
"Un client ne doit pas être forcé de dépendre d'interfaces qu'il n'utilise pas."

Mieux vaut avoir plusieurs petites interfaces spécialisées qu'une grande interface généraliste. Une classe ne devrait pas être obligée d'implémenter des méthodes qu'elle n'utilise pas.

❌ Violation ISP
public interface Travailleur {
  void travailler();
  void manger();
  void dormir();
}

// Un Robot doit implémenter manger() et dormir()
// alors qu'il ne mange ni ne dort !
public class Robot implements Travailleur {
  public void travailler() { /* ok */ }
  public void manger() { /* ??? */ }
  public void dormir() { /* ??? */ }
}
✅ Respect ISP
public interface Travailleur {
  void travailler();
}

public interface EtreVivant {
  void manger();
  void dormir();
}

public class Humain
    implements Travailleur, EtreVivant {
  public void travailler() { ... }
  public void manger() { ... }
  public void dormir() { ... }
}

public class Robot implements Travailleur {
  public void travailler() { ... }
  // Pas forcé d'implémenter manger/dormir
}

✅ Pourquoi c'est important

Des interfaces petites et cohérentes rendent le code plus flexible. On peut combiner les interfaces nécessaires sans être contraint par des méthodes inutiles.
Plusieurs interfaces spécialisées valent mieux qu'une interface fourre-tout — chaque classe n'implémente que ce dont elle a besoin.
D
Dependency Inversion Principle
Principe d'inversion des dépendances
"Les modules de haut niveau ne doivent pas dépendre des modules de bas niveau. Les deux doivent dépendre d'abstractions. Les abstractions ne doivent pas dépendre des détails. Les détails doivent dépendre des abstractions."

En clair : dépendre d'interfaces, pas de classes concrètes. Le code de haut niveau (la logique métier) ne doit pas connaître les détails d'implémentation.

❌ Violation DIP
// Module haut niveau dépend du bas niveau
public class ServiceCommande {
  // Dépend directement de la classe concrète !
  private MySQLDatabase db = new MySQLDatabase();

  public void passerCommande(Commande c) {
    db.sauvegarder(c);
  }
}
// Changer de BDD → modifier ServiceCommande !
✅ Respect DIP
// Abstraction
public interface Database {
  void sauvegarder(Commande c);
}

// Module haut niveau dépend de l'abstraction
public class ServiceCommande {
  private Database db; // interface !

  public ServiceCommande(Database db) {
    this.db = db; // injection de dépendance
  }

  public void passerCommande(Commande c) {
    db.sauvegarder(c);
  }
}
// Changer de BDD → juste une nouvelle classe

✅ Lien avec les Design Patterns

La plupart des Design Patterns respectent DIP — le Strategy dépend de l'interface Strategy (pas de Visa ou Bancontact), le Observer dépend de l'interface Observateur (pas de JeanMichel), l'Abstract Factory dépend des interfaces produit (pas des classes concrètes).
Dépendre d'abstractions (interfaces), jamais de classes concrètes — c'est la base de la flexibilité en POO.
